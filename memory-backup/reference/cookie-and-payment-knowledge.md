# Cookie 与支付流程知识

## Cookie
- **定义**：不是用户凭证本身，而是"容器/机制"。真正的凭证是里面装的 Session ID 或 JWT
- **产生方式**：Cookie 不是浏览器自动生成的，而是网站服务器主动通过 `Set-Cookie` 响应头下发的（或 JS 设置）。纯静态网页可以零 Cookie
- **本质**：网站主动给用户打的"标记/记号"，用于身份标记、行为追踪、偏好记忆
- **临时 Cookie**：相当于游客模式——网站先发个"游客牌"，用户所有操作都挂在这个牌下，登录后"升级"为正式用户
- **一个用户多个 Cookie**：同一网站可设多个键值对（session_id, theme, lang），不同网站各自存各自的，同一用户不同设备各有各的 cookie

### 关键属性
| 属性 | 作用 |
|------|------|
| Name/Value | 存储的键值对 |
| Domain/Path | 限制发送范围 |
| Expires/Max-Age | 不设则关闭浏览器即失效（临时=Session Cookie） |
| HttpOnly | 禁止 JS 读取，防 XSS |
| Secure | 仅 HTTPS 传输 |
| SameSite | 防 CSRF，控制跨站发送 |

## AI 访问网页
- AI 通过 WebFetch 等工具读取前端 HTML/CSS/JS 源码来理解网页内容
- 后端数据库、业务逻辑代码 AI 根本碰不到
- Cookie 对 AI 无意义——收到了当没收到，不维持登录态
- 前端是服务器主动公开的（菜单），后端是锁在服务器里的（后厨菜谱）

## 前后端安全边界
- F12 看 HTML/JS/API 响应 → 合法（服务器主动给你的）
- 通过修改请求、SQL 注入、暴力破解等方式获取非授权数据 → 非法入侵
- 前端挡付费内容（CSS 隐藏/JS 判断）→ 形同虚设，技术上一分钟可绕过
- 正确做法：服务器按付费状态分段返回内容，付费前只给预览，完整内容根本没发到浏览器

## 支付流程（支付宝/微信支付）
- 公司不是去支付宝里开"店铺"上架商品，而是注册成商户拿到 API 密钥
- 支付公司只管钱：谁付了多少、进谁的账户、手续费多少，完全不管你在卖什么
- 商品信息（谁买了什么）存在公司自己网站的数据库里

### 两本账模型
| | 公司网站 | 支付宝/微信 |
|------|--------|--------|
| 记录什么 | 谁买了什么内容 | 谁付了多少钱 |
| 本质 | **业务账** | **资金账** |

两本账通过订单号（order_001 等）对应。

### 支付回调链路
1. 网站生成订单（服务器直连支付宝）
2. 用户跳转支付宝完成付款
3. 用户跳回网站，带临时凭证
4. 网站服务器向支付宝确认付款（服务器对服务器，不经过浏览器）
5. 确认后网站改数据库 → 用户刷新 → Cookie 关联到已付费状态 → 返回完整内容
