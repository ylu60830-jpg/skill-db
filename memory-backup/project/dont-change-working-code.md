## 教训：不要乱改已经正常的代码

**时间:** 2026-06-18

**事件:** 用户说宿舍管理系统前端打不开，实际原因是 MySQL 没启动。启完 MySQL 后，用户已经满意了。但 Claude 自作主张改了一大堆 CSS 和模板（表格滚动、搜索栏布局等），用户表示「之前做的已经挺不错了，我让你解决的也不是这个问题」。

**根本原因:** claude-mem 刚被删除，丢失了昨晚的会话上下文，不知道前端是用户满意的成品。

**规则:**
1. 先确认问题的真正范围，解决完就停
2. 不要「顺便优化」——用户没要求的东西不要改
3. 用户说「打不开」= 解决能打开的问题，不是重新设计
4. 内存缺失时更应该多确认、少假设
5. 改动前先 git diff 看当前状态，了解代码背景

**How to apply:** 每次收到问题报告，先定位根因，解决根因后停下来确认用户是否还需要更多。不要主动扩大范围。
